Jouster Usage Guide

This guide provides detailed information on how to use Jouster in various scenarios. It covers common use cases, command-line options, and best practices.

Command-Line Interface

Jouster provides a command-line interface (CLI) for running tests and managing issues. Here are the available commands:

Run Tests
Run tests and manage issues for failing tests.
npx jouster run-tests [options] [files]

Options:

  • --test-runner <runner>: Specify the test runner to use (default: auto-detect)
  • --generate-issues: Generate issues for failing tests (default: true)
  • --track-issues: Track issues for failing tests (default: true)
  • --close-issues: Close issues for passing tests (default: true)
  • --reopen-issues: Reopen issues for failing tests (default: true)
  • --database-path <path>: Path to the database file (default: test-issue-mapping.json)
  • --tracker-type <type>: Bug tracker type (github or file, default: github)
Setup
Run the setup wizard to configure Jouster for your project.
npx jouster setup
List Issues
List all issues in the mapping database.
npx jouster list-issues

Options:

  • --database-path <path>: Path to the database file (default: test-issue-mapping.json)
  • --status <status>: Filter by issue status (open, closed, all, default: all)
Clean Database
Clean the mapping database by removing entries for tests that no longer exist.
npx jouster clean-database

Options:

  • --database-path <path>: Path to the database file (default: test-issue-mapping.json)
  • --dry-run: Show what would be removed without actually removing anything

Common Use Cases

Running Tests with Jest

npx jouster run-tests --test-runner jest src/__tests__

Running Tests with Mocha

npx jouster run-tests --test-runner mocha --mocha-ui bdd test/**/*.js

Running Tests with a Custom Configuration

npx jouster run-tests --config ./jouster.config.js

Running Tests Without Generating Issues

npx jouster run-tests --no-generate-issues

Running Tests Without Closing Issues

npx jouster run-tests --no-close-issues

Running Tests with a Different Database Path

npx jouster run-tests --database-path ./custom-mapping.json

Running Tests with File Tracker

npx jouster run-tests --tracker-type file --bugs-dir ./bugs

Continuous Integration

Jouster can be integrated into your CI/CD pipeline to automatically create and manage issues for failing tests. Here's how to set it up:

GitHub Actions

# .github/workflows/tests.yml
name: Tests

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v2
    
    - name: Set up Node.js
      uses: actions/setup-node@v2
      with:
        node-version: '16'
        
    - name: Install dependencies
      run: npm ci
      
    - name: Run tests with Jouster
      run: npx jouster run-tests
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

CircleCI

# .circleci/config.yml
version: 2.1
jobs:
  test:
    docker:
      - image: cimg/node:16.13
    steps:
      - checkout
      - run:
          name: Install dependencies
          command: npm ci
      - run:
          name: Run tests with Jouster
          command: npx jouster run-tests
          environment:
            GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
workflows:
  version: 2
  build-and-test:
    jobs:
      - test

Travis CI

# .travis.yml
language: node_js
node_js:
  - 16
script:
  - npm ci
  - npx jouster run-tests

Environment Variables

Jouster can be configured using environment variables. These variables override the configuration in the jouster.config.js file.

# Whether to generate issues for failing tests
GENERATE_ISSUES=true

# Whether to track issues for failing tests
TRACK_ISSUES=true

# Whether to close issues for passing tests
CLOSE_ISSUES=true

# Whether to reopen issues for failing tests
REOPEN_ISSUES=true

# Path to the database file
DATABASE_PATH=test-issue-mapping.json

# Default labels for issues
DEFAULT_LABELS=bug,test-failure

# Bug tracker type ('github' or 'file')
TRACKER_TYPE=github

# Path to the bugs directory (for file tracker)
BUGS_DIR=./bugs

# GitHub labels to apply to issues
GITHUB_LABELS=bug,test-failure

# Whether to use the REST API instead of the CLI
GITHUB_USE_REST=false

# GitHub personal access token (required when using REST API)
GITHUB_TOKEN=your-token

# GitHub repository in the format 'owner/repo' (required when using REST API)
GITHUB_REPO=owner/repo

# Path to the template directory
TEMPLATE_DIR=./templates

Best Practices

1. Use a Dedicated Branch for Test Fixes

When fixing failing tests, create a dedicated branch for each issue. This makes it easier to track changes and ensures that each fix is properly reviewed.

2. Include Issue Numbers in Commit Messages

When fixing a failing test, include the issue number in the commit message. This helps to track which commits are related to which issues.

git commit -m "Fix failing test for Calculator.subtract (fixes #123)"

3. Use Custom Templates for Different Test Types

Create custom templates for different types of tests. This makes it easier to understand the context of a failing test and helps to prioritize fixes.

4. Clean the Database Regularly

Run the clean-database command regularly to remove entries for tests that no longer exist. This keeps the database clean and prevents issues from being reopened for tests that have been removed.

npx jouster clean-database

5. Use a Consistent Naming Convention for Tests

Use a consistent naming convention for tests. This makes it easier to identify tests and helps Jouster to track issues more effectively.

Troubleshooting

Issues Not Being Created

  • Check that the GitHub CLI is installed and authenticated
  • Check that the GitHub token has the necessary permissions
  • Check that the repository exists and is accessible
  • Check that the test runner is configured correctly

Issues Not Being Closed

  • Check that the close-issues option is enabled
  • Check that the GitHub CLI is installed and authenticated
  • Check that the GitHub token has the necessary permissions
  • Check that the issue exists and is open

Issues Not Being Reopened

  • Check that the reopen-issues option is enabled
  • Check that the GitHub CLI is installed and authenticated
  • Check that the GitHub token has the necessary permissions
  • Check that the issue exists and is closed

Database Errors

  • Check that the database file exists and is writable
  • Check that the database file is valid JSON
  • Try running the clean-database command to remove invalid entries
