Contributing to Jouster

Thank you for your interest in contributing to Jouster! This page provides guidelines and information for contributing to the project.

Code of Conduct

We expect all contributors to follow our Code of Conduct. By participating in this project, you agree to abide by its terms.

Our Pledge

In the interest of fostering an open and welcoming environment, we as contributors and maintainers pledge to making participation in our project and our community a harassment-free experience for everyone, regardless of age, body size, disability, ethnicity, gender identity and expression, level of experience, nationality, personal appearance, race, religion, or sexual identity and orientation.

Our Standards

Examples of behavior that contributes to creating a positive environment include:

  • Using welcoming and inclusive language
  • Being respectful of differing viewpoints and experiences
  • Gracefully accepting constructive criticism
  • Focusing on what is best for the community
  • Showing empathy towards other community members

Examples of unacceptable behavior by participants include:

  • The use of sexualized language or imagery and unwelcome sexual attention or advances
  • Trolling, insulting/derogatory comments, and personal or political attacks
  • Public or private harassment
  • Publishing others' private information, such as a physical or electronic address, without explicit permission
  • Other conduct which could reasonably be considered inappropriate in a professional setting

Getting Started

Fork the Repository

To contribute to Jouster, you'll need to fork the repository:

  1. Go to the Jouster repository on GitHub
  2. Click the "Fork" button in the top-right corner
  3. Clone your fork to your local machine: git clone https://github.com/YOUR_USERNAME/jouster.git
  4. Add the original repository as a remote: git remote add upstream https://github.com/TSavo/jouster.git

Set Up the Development Environment

To set up the development environment:

  1. Install Node.js (version 14 or later)
  2. Install dependencies: npm install
  3. Build the project: npm run build
  4. Run tests: npm test

Development Workflow

Branching Strategy

We use a simple branching strategy:

  • main: The main branch contains the latest stable release
  • develop: The development branch contains the latest development changes
  • Feature branches: Created from develop for new features
  • Bugfix branches: Created from develop for bug fixes

Creating a Branch

To create a new branch:

git checkout develop
git pull upstream develop
git checkout -b feature/your-feature-name

Making Changes

When making changes:

  • Follow the code style of the project
  • Write tests for your changes
  • Make sure all tests pass: npm test
  • Update documentation if necessary

Committing Changes

When committing changes:

  • Use descriptive commit messages
  • Reference issue numbers in your commit messages: Fix #123: Add support for XYZ
  • Make small, focused commits rather than large, sweeping changes

Submitting a Pull Request

To submit a pull request:

  1. Push your branch to your fork: git push origin feature/your-feature-name
  2. Go to the Jouster repository on GitHub
  3. Click the "New pull request" button
  4. Select your branch and fill out the pull request template
  5. Submit the pull request

Coding Standards

Code Style

We use ESLint and Prettier to enforce code style. To check your code style:

npm run lint

To automatically fix code style issues:

npm run lint:fix

TypeScript

We use TypeScript for type safety. Make sure your code passes the TypeScript compiler:

npm run build

Testing

We use Jest for testing. Make sure your code has tests and all tests pass:

npm test

To run tests with coverage:

npm run test:coverage

Documentation

We use JSDoc for code documentation. Make sure your code is well-documented:

/**
 * Adds two numbers.
 * @param {number} a - The first number.
 * @param {number} b - The second number.
 * @returns {number} The sum of a and b.
 */
function add(a, b) {
  return a + b;
}

Pull Request Guidelines

Pull Request Template

When submitting a pull request, please use the provided template and fill out all sections:

# Description

Please include a summary of the change and which issue is fixed. Please also include relevant motivation and context.

Fixes # (issue)

## Type of change

Please delete options that are not relevant.

- [ ] Bug fix (non-breaking change which fixes an issue)
- [ ] New feature (non-breaking change which adds functionality)
- [ ] Breaking change (fix or feature that would cause existing functionality to not work as expected)
- [ ] This change requires a documentation update

## How Has This Been Tested?

Please describe the tests that you ran to verify your changes. Provide instructions so we can reproduce. Please also list any relevant details for your test configuration.

- [ ] Test A
- [ ] Test B

## Checklist:

- [ ] My code follows the style guidelines of this project
- [ ] I have performed a self-review of my own code
- [ ] I have commented my code, particularly in hard-to-understand areas
- [ ] I have made corresponding changes to the documentation
- [ ] My changes generate no new warnings
- [ ] I have added tests that prove my fix is effective or that my feature works
- [ ] New and existing unit tests pass locally with my changes
- [ ] Any dependent changes have been merged and published in downstream modules

Review Process

All pull requests will be reviewed by at least one maintainer. The review process includes:

  • Code review: Checking code quality, style, and correctness
  • Test review: Ensuring tests are adequate and pass
  • Documentation review: Ensuring documentation is updated

Continuous Integration

All pull requests are automatically tested by our continuous integration system. Make sure all tests pass before submitting a pull request.

Issue Guidelines

Bug Reports

When reporting a bug, please include:

  • A clear and descriptive title
  • Steps to reproduce the bug
  • Expected behavior
  • Actual behavior
  • Screenshots (if applicable)
  • Environment information (OS, Node.js version, etc.)

Feature Requests

When requesting a feature, please include:

  • A clear and descriptive title
  • A detailed description of the feature
  • Why the feature would be useful
  • How the feature would work
  • Examples of similar features in other projects (if applicable)

Questions

If you have a question about Jouster, please:

  • Check the documentation first
  • Search for existing issues
  • If you can't find an answer, create a new issue with the "question" label

Community

Communication Channels

We use the following channels for communication:

  • GitHub Issues: For bug reports, feature requests, and questions
  • GitHub Discussions: For general discussion and community interaction
